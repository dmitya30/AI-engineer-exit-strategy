**Связь с Дмитрием:** старый личный контакт, общение на "ты"

## 2. Задача

Единая точка приёма заявок - сайт. Клиент с QR (визитка, реклама на машине) заходит, видит позиционирование + расписание по городам, оставляет заявку. Заявка падает Дмитрию... (Елене) в Telegram + в таблицу. Дальше - ручная переписка в удобном клиенту канале.

**Что осознанно НЕ делаем:**
❌ Ботов в мессенджерах (Telegram/WhatsApp/Max/ВК) - дорого, хрупко, не закрывает расписание по городам
❌ WhatsApp-автоматизацию - нелегально на личном аккаунте + RU-блокировки звонков с 08.2025, риск полной блокировки
❌ AI-агента в виджете на старте - риск галлюцинаций про услуги/цены, избыточно при потоке 2-3/день
❌ CRM на MVP - таблица + Telegram закрывают всё
❌ Жёсткое бронирование слотов - нужен только "календарь доступности" (ориентировочная запись, финал стыкуется вручную)
❌ Привязку к конструктору Т-Банка - делаем свой контур на VPS

## 3. Решение и этапы

📌 **Этап 1. Сайт + приём заявок** - `30 000`, срок 1-1,5 нед.
Лендинг (1 стр.), форма-квалификатор как пошаговый диалог (мгновенные ответы, без LLM), сбор данных (был/не был, индивид/компания, даты, предпочтительный канал связи + контакт), заявка в Telegram + таблицу, домен/хостинг/QR.

📌 **Этап 2. Расписание по городам + панель** - `35 000`, срок 1-2 нед.
Self-service панель для Елены (правит расписание сама), отображение "город + доступные дни" клиенту, запись с привязкой к городу/дате, Telegram-напоминание об истечении расписания. Запускается под выезд на Алтай ("скажет еду - стартую").

✅ **Юридика 152-ФЗ** - бесплатно (вынесена из цены)
Шаблоны (политика, согласие, оферта) бесплатно, подачу в РКН клиент делает сам на pd.rkn.gov.ru. Дмитрий не юрист - позиционируется как "рабочий минимум", не аудит.

⏩ **На потом (след. год, отдельно):** Яндекс.Метрика + сбор аудитории по РФ под Алтай. Реклама на машине у клиента уже есть (QR делаем в Этапе 1).

## 5. Техническая архитектура (для Д4)

**Стек:** свой бэкенд на VPS (Python/FastAPI или Node), статичный фронт с виджетом-формой, хранилище заявок (SQLite или Google Sheets), уведомления через Telegram Bot API. n8n НЕ используется (избыточен для задачи).
**Внешние зависимости:** только Telegram Bot API (легально, стабильно).
**Критичное требование:** дублирование заявок (таблица/БД + Telegram-пуш) - не полагаться только на пуш, иначе пролистанная заявка = потерянный клиент.
**Виджет:** пошаговый сценарий на кнопках/полях, оформленный как живой диалог. БЕЗ LLM на MVP. AI-слой - возможный этап 3 (+`20-30К`), только при росте потока.
**Расписание (Этап 2):** self-service панель (выбран клиентом из 3 вариантов - предпочёл полный визуальный контроль без посредников-таблиц), модель "календарь доступности" read-only для клиента, не бронирование слотов.

## 7. Статус и следующие шаги (ОБНОВЛЕНО 2026-06-07)

**Статус:** 🔴 Архив. Отказ по таймингу (НЕ по качеству).

**Хронология финала:**
КП отправлено -> прочитано сразу -> молчание неделю -> пуш (открытый вопрос "что останавливает") -> клиент ответил развёрнуто и честно. Решение: остаётся на бесплатном конструкторе Т-Банка + ручное ведение. Причина - фаза чистых вложений (закупка инвентаря: простыни, полотенца, тематическая одежда/вышивка), проект ещё не окуплен, бюджет на сайт/бота выделить не может. Прямая цитата клиента: *"как начну развиваться, так и дело дойдёт до правильного сайта и бота, сейчас не могу выделить бюджет"*.

**Реакция Дмитрия:** без дожима. Ответ: *"Понял. Как надумаешь, будешь готова, пиши."* Контакт сохранён тёплым, дверь открыта со стороны клиента.

**Тип отказа:** отложенный спрос + вероятно несуществующий спрос (см. блок 9). НЕ заносить в "проиграл по качеству" - это искажает статистику. Категория: "не та фаза бизнеса".

**Точка возможного возврата:** инициатива на стороне клиента. Со стороны Дмитрия - не пушить продажу. Допустимый тёплый ход (не обязательный): бесплатно глянуть её ТБ-конструктор, когда соберёт, дать пару советов по ручному приёму заявок. Держит контакт без продажи.

## 9. Инсайты для калибровки квалификации (НОВЫЙ БЛОК)

Главный вывод кейса: аудит и решение были технически верными, но продавались не тому моменту и, вероятно, не тому типу клиента. Сигналы были видны с первого голосового ("работаю только наёмно", "на личный телефон заявок ноль"), но не доведены до решения на этапе диагностики.

Две калибровки на будущее, обе - ДО проектирования решения:

📌 **Квалификация по фазе бизнеса.** Фаза "чистых вложений, потока нет, проект не окуплен" = не продавать инфраструктуру/автоматизацию. Нечего автоматизировать. На этой фазе клиенту нужны первые клиенты, а не система приёма несуществующего потока. Маркер: клиент перечисляет затраты на запуск, а не доходы.

📌 **Квалификация по типу личности.** Ремесленник vs растущий предприниматель. Ремесленник проговаривает рост абстрактно ("потом, как разовьюсь, может франшиза"), без конкретики и без боли от текущего потолка - ему комфортно с 2-3 клиентами. Растущий говорит о конкретной боли ("не хватает рук на спрос, теряю деньги здесь"). Нет боли от потолка = нет покупки, любой оффер бесполезен. Лена - первый тип: рост обсуждается, но боли от 2-3 клиентов нет.

Практика: на диагностике явно проверять оба пункта. Вопрос-маркер - не "хочешь расти?" (все скажут да), а "что тебе сейчас мешает / чего не хватает прямо сейчас?". Отсутствие конкретного болевого ответа = клиент не готов, продавать систему рано или не нужно вовсе.
