# Один + ИИ-команда: архитектура усиления на 2026

Цель этого ответа — дать тебе **реальную карту**: что в 2026 году действительно делегируется агентам, что нет, какие стеки работают, как не сжечь токены и не уйти в минус по операционке. Это не «обзор технологий», это **выбор твоего рабочего стека** на ближайший год.

В конце — конкретные рекомендации по твоему сетапу.

## Базовое разделение труда

Прежде чем говорить о технологиях, нужно понять принцип: **что делает человек, что делает агент**. Это не догма, это рабочая модель 2026 года, проверенная индустрией.

**Человек делает:**
1. **Стратегические решения** (что строим, для кого, какая модель монетизации)
2. **Принятие новой информации** (новый клиент, новый рынок, новое возражение — нужно понять и решить)
3. **Финальные коммуникации в высоких ставках** (закрытие сделки, разруливание конфликта, разговор с ЛПР)
4. **Креативные прорывы** (новая идея продукта, нестандартный ход, разворот стратегии)
5. **Этические решения и контроль качества**
6. **Обучение и калибровка агентов** (промпты, примеры, корректировка)

**Агенты делают:**
1. **Рутинные операции** в известных сценариях (парсинг, классификация, обработка данных)
2. **Первичные диалоги** по проверенным сценариям (квалификация, FAQ, базовая поддержка)
3. **Кодинг по чёткому ТЗ** (90% реализации, человек ревьюит и корректирует)
4. **Контент-генерацию по шаблону** (тексты, посты, ответы по структуре)
5. **Мониторинг и алертинг** (отслеживание триггеров, эскалация)
6. **Документирование и репортинг**

**Серая зона (зависит от уровня модели и зрелости промптов):**
1. Принятие решений в нестандартных ситуациях
2. Эмоционально нагруженные коммуникации
3. Сложный анализ с длинной цепочкой рассуждений
4. Адаптация к меняющемуся контексту

В 2026 году с Claude Opus 4.7 серая зона **сильно сместилась в сторону агентов**. То, что в 2024-м было «только человек», в 2026-м — «агент с проверкой человеком». Но **не полностью без человека**.

## Стек на 2026: что выбрать

Я не буду давать каталог всех инструментов. Дам **рекомендуемый стек под твою задачу** с обоснованием каждого выбора.

### Кодинг и разработка софта (мета-уровень — ты строишь свой инструмент)

**Claude Code** — однозначно. Это командная-строковый агент, который работает с кодовой базой, может читать/писать/рефакторить, понимает контекст репозитория. Для разработчика твоего профиля это **главное усиление**. С его помощью реалистично собрать MVP лидген-системы за 2–4 недели в одиночку.

**Почему именно он, а не Cursor/Windsurf/Continue:** Claude Code оптимизирован под Claude Opus 4.7 нативно, лучше работает с длинным контекстом, у него лучшая работа с агентскими цепочками (планирование → разбиение → выполнение). Cursor сильнее как IDE-помощник для парного программирования, но Claude Code сильнее как автономный исполнитель задач.

**Практический режим:** ты ставишь задачу высокого уровня («собери модуль парсинга Telegram-каналов с фильтрацией по активности»), Claude Code разбивает на подзадачи, пишет, тестирует, показывает результат. Ты ревьюишь. Этот режим экономит **70–85% твоего времени** на типовых задачах.

### Экономия токенов в Claude Code

Это твой главный страх, я его слышу. Дам конкретные правила.

**Правило 1: контекстная гигиена.** Не загружай весь репозиторий, если задача касается одного модуля. Claude Code умеет работать с указанными файлами — указывай явно. Сжигание контекста идёт не от объёма работы, а от того, что модель «таскает» лишний контекст из сессии в сессию.

**Правило 2: чёткие итерации.** Лучше 5 коротких задач с конкретным результатом, чем одна большая «сделай всю систему». Каждая задача — отдельная сессия, отдельный контекст.

**Правило 3: гибрид моделей.** Не вся работа требует Opus. Простую генерацию (форматирование текста, базовая классификация, парсинг шаблонов) делай через Haiku 4.5 — он в 5–10 раз дешевле и для этих задач достаточен. **Opus только там, где нужен Opus** (сложная архитектура, нестандартные диалоги, debug сложных проблем). Это снижает счёт в 3–5 раз.

**Правило 4: кэширование промптов.** Anthropic API поддерживает prompt caching — стабильные части промпта (системный промпт, инструкции, базы знаний) кэшируются и оплачиваются по сильно сниженной ставке. При правильной структуре промптов экономия 40–70%.

**Правило 5: батчинг.** Если делаешь массовую классификацию (например, 1000 источников), не дёргай API по одному — батчи по 10–50 объектов в одном запросе. Снижение расходов в 3–5 раз.

При соблюдении этих правил **месячный бюджет на разработку твоего лидген-софта** через Claude Code реалистично укладывается в 5–15к рублей. Операционная работа системы (которая после запуска бежит сама) — отдельный бюджет, я к нему ниже вернусь.

### MCP (Model Context Protocol) — нужен ли тебе сейчас

MCP — это протокол, позволяющий Claude (и другим LLM) подключаться к внешним системам: базам данных, файловым системам, веб-сервисам, твоим собственным API. По сути — стандартизированный способ давать модели «инструменты».

**Для твоей задачи MCP полезен в двух местах:**

Первое — **в твоей разработке**. Claude Code через MCP может напрямую работать с твоей БД лидов, читать логи системы, дергать твои API для тестирования. Это резко ускоряет цикл разработки.

Второе — **в твоей продакшен-системе**. Твой dialog-агент через MCP может ходить в CRM, в базу знаний, в логи прошлых разговоров. Это превращает агента из «генератора текстов» в «оператора с инструментами».

**Когда осваивать.** Не на этапе MVP. Сначала запусти систему «прямо», без MCP. Когда увидишь, что определённые операции повторяются и тебе хочется дать агенту больше автономности — тогда добавляй MCP-серверы под конкретные нужды. Это 1–2 дня на освоение для твоего уровня.

**Что не делать.** Не строй сразу «всё через MCP» — это over-engineering. MCP — инструмент решения конкретной задачи, а не архитектурная догма.

### RAG: где применять, где не применять

Ты уже знаком, поэтому коротко. RAG нужен **не везде**.

**Применять обязательно:**
- В dialog-агенте, который ведёт длинные разговоры с лидами (история разговора + контекст лида)
- В консультанте-боте (база знаний по теме: банкротство, AI-внедрение, что угодно)
- В обучении системы на твоих успешных кейсах (примеры хороших ответов)

**Не применять:**
- Для разовой классификации (зачем RAG, если задача — «это лид или не лид»?)
- Для генерации шаблонного контента
- Для парсинга и обработки данных

**Простое правило:** RAG = когда модели нужны **знания**, которых нет в её весах. Если задача про знания из общего интеллекта — не нужен.

### Оркестрация: n8n vs собственный код

У тебя уже опыт в n8n, и это ценно. Дам честное сравнение.

**n8n подходит для:**
- Быстрого прототипирования воркфлоу
- Интеграций с внешними сервисами (API всего на свете уже подключено)
- Расписаний и триггеров
- Сценариев, где нужен визуальный обзор

**n8n плох для:**
- Долгоживущих процессов с состоянием (когда сессия диалога должна жить днями/неделями)
- Сложной асинхронной логики
- Высоких нагрузок (тысячи операций в секунду)
- Тонкой работы с памятью и контекстом агента

**Моя рекомендация для тебя:**
- MVP до 100 лидов/день — на n8n. Не трать время на свой код там, где есть готовое
- Когда система разрастётся (300+ лидов/день, сложные диалоги) — переносить ядро (dialog-агент, состояние лидов) на Python/FastAPI, оркестрацию вокруг оставить на n8n. Это гибрид, который работает лучше всего

### Память и состояние

Это часто упускают, а здесь — критичный момент. Агент без памяти = плохой агент.

**Слои памяти, которые нужны:**
1. **Краткосрочная** (в рамках одной сессии диалога) — стандартная функция любого LLM API, ничего особого делать не надо
2. **Долгосрочная по лиду** (вся история взаимодействия с конкретным человеком) — RAG + БД, обращаться при каждом новом контакте
3. **Семантическая по нише** (твоя экспертиза, кейсы, FAQ) — векторная БД, общая для всех агентов
4. **Эпизодическая системная** (что сработало, что не сработало, паттерны) — это твоя «обучающая» база. Каждое успешное закрытие → пример для следующих сессий

Без слоя 4 система не учится. Это означает, что **ты вручную** добавляешь успешные кейсы в базу, чтобы агенты на них учились.

## Реальный «штат» для одного человека с AI-усилением

Давай я переведу это в метафору «штатного расписания», чтобы было видно, **сколько ты экономишь**.

В классическом лидген-агентстве для выполнения той же работы нужны:
- 1 sales development representative (SDR) — выходит на холодные источники, ~80–150к/мес ЗП
- 1 копирайтер для outreach-сообщений и контента — 60–120к/мес
- 1 lead researcher — 50–100к/мес
- 1 CRM-оператор — 40–80к/мес
- 0,5 разработчика для поддержки — 100–200к/мес условно
- 0,3 руководителя — 80–150к/мес условно

**Итого фонд:** ~400–800к/мес для команды среднего качества.

**Твой сетап:**
- Ты сам = стратег + ЛПР-коммуникатор + ревьюер + разработчик
- Discovery-агент + Qualification-агент + Enrichment-агент = функция researcher
- First-touch агент = функция SDR на холодных
- Dialog-агент = функция SDR на тёплых + копирайтер
- CRM-автоматизация = функция оператора
- Claude Code = функция разработчика на поддержке

**Стоимость:** API + инфраструктура = 10–30к/мес на полнофункциональную систему обработки 100–300 лидов/день.

**Это 15–30-кратная экономия по фонду** при сопоставимом качестве работы (с поправкой на то, что ты сам — узкое место в «человеческой» части).

## Где находится потолок «одного + AI-команды»

Это нужно проговорить, чтобы не было иллюзий.

**Реалистичный потолок одного человека с AI-усилением:**
- 5–15 активных клиентов в обслуживании одновременно (как лидген-агентство)
- 200–500 лидов в воронке параллельно
- Доход 300–800к/мес в зависимости от модели и ниши

**Дальше — потолок.** Когда выходишь за это — нужны люди. Не потому что технически нельзя, а потому что:
- Сам ты — bottleneck в коммуникациях с клиентами
- Качество страдает на масштабе без человеческого ревью
- Этическая и репутационная экспозиция растёт нелинейно

Для твоей цели «100к/мес» этого с большим запасом хватает. Цель достижима с 1–3 активными клиентами или одной хорошо работающей партнёрской воронкой.

## Конкретный технологический выбор для твоего MVP

Соберу всё в один сетап, по которому можно начинать.

**Языки и фреймворки:**
- Python 3.11+ (асинхронная обработка через asyncio)
- FastAPI для собственных API
- Pyrogram для Telegram User API
- Playwright для парсинга «сложных» страниц
- SQLAlchemy + PostgreSQL для основной БД

**AI-слой:**
- Anthropic API (Claude Opus 4.7 для ключевых задач, Haiku 4.5 для рутины)
- Prompt caching включён везде, где применимо
- LangChain или прямой API — на твой вкус (LangChain даёт удобство, прямой API даёт контроль)

**RAG:**
- Qdrant как векторная БД (легковесный, локально работает) или Chroma для совсем простых случаев
- Embeddings от OpenAI (text-embedding-3-small) — дёшево и качественно

**Оркестрация:**
- n8n для воркфлоу высокого уровня (расписания, триггеры, простые цепочки)
- Собственный Python-сервис для dialog-агента и сложной логики

**Инфраструктура:**
- VPS у Selectel/Timeweb для РФ — 1500–3000 руб./мес за достаточный сервер
- Резидентные прокси: Proxy6.net, Astroproxy — 1000–3000 руб./мес
- Telegram-аккаунты: 3–5 для MVP, ротация

**Мониторинг:**
- Grafana + Prometheus для метрик системы (если хочется красиво)
- Просто Telegram-бот, шлющий тебе алерты — для MVP достаточно

**Разработка:**
- Claude Code как основной инструмент
- Git/GitHub для кода
- Локальная разработка → деплой на VPS

**Бюджет MVP:**
- Разовый закуп (аккаунты, домены, начальная инфраструктура): 5–10к
- Месячная операционка (хостинг, прокси, API в режиме разработки): 8–15к
- Месячная операционка после запуска (API в продакшене): 15–30к в зависимости от нагрузки

При выходе на 100к+ дохода — это нормальная маржа.

## Что я рекомендую делать прямо сейчас

Сухо и конкретно:

1. **Создай GitHub-репозиторий** для своего лидген-софта (даже приватный — потом будет история). Установи Claude Code локально, поработай с ним 1–2 часа, чтобы освоиться.

2. **Разверни базовый каркас за 2–3 дня:** Python-проект, БД, базовая схема для лидов и источников. Это фундамент.

3. **Сделай парсер Telegram (Pyrogram)** — основная функция: «дай мне посты из канала X за последние 24 часа». Через Claude Code это 1 день.

4. **Сделай qualification-агент** — простой промпт «вот пост, вот контекст, является ли это лидом по критериям N». Через Opus 4.7. Полдня.

5. **Тестируй на твоей нише** (выбери одну — B2B или банкротство). Прогоняй через систему 2–3 канала вручную, смотри на качество классификации.

6. **Параллельно** — продолжай Ответ 2 (заработок сейчас). Не уходи в разработку с головой, иначе через месяц у тебя будет крутой каркас и ноль денег.
